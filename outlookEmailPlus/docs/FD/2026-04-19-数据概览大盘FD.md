## 1. 功能定义

### 1.1 核心功能

将现有前端 `page-dashboard`（仪表盘页）全面替换为五 Tab 的「数据概览大盘」，聚合展示系统运营数据：

1. **总览**：账号状态分布快照 + 邮箱池实时状态 + Token 刷新健康 + 今日快捷数字
2. **验证码提取**：近 7 天提取效率统计（需新增提取日志持久化能力）
3. **对外 API**：调用方统计、端点分布、日调用趋势（纯 CSS 柱图）
4. **邮箱池**：Claim/Complete/Release/Expire 操作分布 + 项目维度统计
5. **系统活动**：审计操作分布 + 通知推送健康 + 系统活动时间线

**术语对齐（按当前实际实现）**：

- 本文中的「统一监控面板」指主应用内的 `page-dashboard` 数据概览页。
- 本次用户反馈里的「正常前端 UI 提取验证码」指主应用页面里的提取按钮；当前主要由 `static/js/features/groups.js` 调用 `/api/emails/<email>/extract-verification`，临时邮箱则调用 `/api/temp-emails/<email>/extract-verification`。
- 统一监控面板的验证码相关统计读取自 `verification_extract_logs`，因此要先确认具体提取链路有没有真的写入这张表，不能只看前端有没有触发重新拉取。

## 2. 数据模型设计

### 2.1 新增表：`verification_extract_logs`（DB v23）

```sql
CREATE TABLE IF NOT EXISTS verification_extract_logs (
    id             INTEGER PRIMARY KEY AUTOINCREMENT,
    account_id     INTEGER NOT NULL,
    channel        TEXT    NOT NULL,          -- graph_inbox / graph_junk / imap_new / imap_old / temp_mail / ai_fallback
    started_at     REAL    NOT NULL,          -- Unix timestamp（秒，含毫秒小数）
    finished_at    REAL    NOT NULL,          -- Unix timestamp（秒，含毫秒小数）
    duration_ms    INTEGER NOT NULL,          -- finished_at - started_at 取整到毫秒
    result_type    TEXT    NOT NULL,          -- code / link / none
    code_found     TEXT,                      -- 提取到的验证码或链接，未提取时为 NULL
    used_ai        INTEGER NOT NULL DEFAULT 0, -- 是否触发 AI fallback（0/1）
    error_code     TEXT,                      -- 失败时错误码，成功时为 NULL
    trace_id       TEXT,                      -- 关联 trace_id（可选）
    created_at     REAL    NOT NULL DEFAULT (unixepoch('now'))
);
CREATE INDEX IF NOT EXISTS idx_vel_account_id     ON verification_extract_logs (account_id);
CREATE INDEX IF NOT EXISTS idx_vel_started_at     ON verification_extract_logs (started_at);
CREATE INDEX IF NOT EXISTS idx_vel_channel        ON verification_extract_logs (channel);
CREATE INDEX IF NOT EXISTS idx_vel_result_type    ON verification_extract_logs (result_type);
```

**字段说明**：

| 字段 | 类型 | 说明 |
|------|------|------|
| `account_id` | INTEGER | 正数表示 `accounts.id`；负数表示 `temp_emails.id` 的哨兵映射（写入时取负值），允许源对象删除后记录保留 |
| `channel` | TEXT | 实际触发提取的通道（非路由计划中的通道）|
| `started_at` / `finished_at` | REAL | 浮点秒（`time.time()`），精度到毫秒 |
| `duration_ms` | INTEGER | 端到端耗时（毫秒），含重试但不含等待轮询时间 |
| `result_type` | TEXT | `code`=提取到验证码，`link`=提取到链接，`none`=未找到 |
| `used_ai` | INTEGER | 是否触发了 AI 识别兜底（`1`=是）|
| `error_code` | TEXT | `AUTH_EXPIRED` / `NETWORK_ERROR` 等，成功时为 `NULL` |

> 实际落地时，为兼容既有测试数据准备语句，还同步补齐了以下兼容字段：`audit_logs.operator`、`audit_logs.status`、`external_api_consumer_usage_daily.caller_id`、`external_api_consumer_usage_daily.date`、`external_api_consumer_usage_daily.call_count`、`account_claim_logs.claimed_at`。

### 2.2 现有可复用表

| 表名 | 用途 | 关键字段 |
|------|------|---------|
| `accounts` | 账号状态 | `pool_status`, `status`, `last_used_at` |
| `refresh_runs` | Token 刷新批次 | `started_at`, `finished_at`, `success_count`, `fail_count` |
| `external_api_consumer_usage_daily` | 外部调用方日级统计 | `date`, `caller_id`, `key_name`, `request_count`, `success_count` |
| `account_claim_logs` | 邮箱池操作日志 | `action`, `caller_id`, `project_key`, `result`, `claimed_at` |
| `account_project_usage` | 项目维度成功记录 | `project_key`, `caller_id`, `account_id` |
| `audit_logs` | 审计操作日志 | `action`, `created_at`, `ip` |
| `notification_delivery_logs` | 通知投递记录 | `channel`, `status`, `created_at` |

### 4.2 JS 模块：`static/js/features/overview.js`

新建模块，职责：

| 函数 | 说明 |
|------|------|
| `initOverview()` | 初始化：注册 Tab 切换、绑定刷新按钮 |
| `switchOverviewTab(tabId)` | 切换 Tab，懒加载该 Tab 数据 |
| `loadOverviewTab(tabId)` | 按 Tab 统一调度接口请求与渲染 |
| `fetchOverviewData(tabId)` | 请求对应 `/api/overview/*` 接口 |
| `renderDataCard(options)` | 统一渲染玻璃数据卡片（标题 / badge / hover note / body） |
| `renderHoverNote(text)` | 渲染卡片级自定义悬浮说明层 |
| `renderBarChart(container, data)` | 纯 CSS 柱图渲染（Tab 3 专用）|
| `renderTimeline(container, events)` | 时间线渲染（Tab 5 专用）|

**加载策略**：
- 进入 `page-dashboard` 时自动加载当前 Tab，并强制向后端重新拉取一次，保证展示的是数据库最新值
- 其余 Tab 在首次点击时懒加载，加载后缓存结果
- 点击「刷新」按钮清空所有 Tab 缓存并重新加载当前 Tab
- 当验证码提取成功后，前端会发出 `overview-data-changed` 事件，失效 `summary` / `verification` / `activity` 缓存；如果概览页当前可见，则立即重拉当前 Tab

**当前刷新边界（按实际代码）**：

1. 当前概览页会因为以下三类触发重新拉取：手动点击刷新按钮、重新进入 dashboard 时的强制拉取、主应用页面内成功提取后的 `overview-data-changed` 事件。
2. `overview-data-changed` 负责让前端缓存失效并重新请求 overview 接口；主应用前端 UI 现在已经在提取成功后同时具备“写日志 + 重拉”两件事。
3. 普通账号前端提取已统一复用 `external_api.py:get_verification_result()`；临时邮箱提取则复用同一张 `verification_extract_logs` 表，并用负数 `temp_emails.id` 作为 `account_id` 哨兵值。补充约束：主应用 Web 提取接口继续保留原有内部错误语义（如 `EMAIL_NOT_FOUND` / `IMAP_AUTH_FAILED`），不直接复用 external API 的对外错误码口径。
4. 因此，当前主应用前端 UI 的普通账号与临时邮箱提取，都会进入统一监控面板；浏览器 AIUI / 对外 API 仍然依赖面板的下一次重新拉取来显示最新值。

## 5. 验证码提取埋点设计

### 5.1 埋点位置

在 `outlook_web/services/external_api.py:get_verification_result()` 最外层包裹埋点，并由 `verification_channel_routing.py` 透传 `_log_channel`：

```python
# 伪代码示意
started_at = time.time()
try:
    result = do_extract(...)
    finished_at = time.time()
    _write_extract_log(
        account_id=account_id,
        channel=actual_channel,
        started_at=started_at,
        finished_at=finished_at,
        result_type=result.result_type,  # "code" / "link" / "none"
        code_found=result.value,
        used_ai=result.used_ai,
        error_code=None,
        trace_id=trace_id,
    )
except Exception as e:
    finished_at = time.time()
    _write_extract_log(
        ...,
        result_type="none",
        code_found=None,
        used_ai=False,
        error_code=classify_error(e),
        trace_id=trace_id,
    )
    raise
```

### 5.2 `_write_extract_log` 实现要点

- 写入到新增的 `verification_extract_logs` 表
- **失败安全**：采用 `try/except` 吞掉写入失败，不影响主流程
- `duration_ms` = `int(round((finished_at - started_at) * 1000))`
- 当触发了 AI fallback 且成功时，`channel` 记录为 `ai_fallback`，`used_ai=1`

> **按当前代码补充说明**：普通账号前端提取已接入 `external_api.py:get_verification_result()`；临时邮箱提取复用同一张日志表，写入时采用负数 `temp_emails.id` 作为 `account_id` 哨兵值，overview 查询层会再把它映射回临时邮箱地址。与此同时，主应用 Web 提取接口依然保留旧的内部错误语义与兼容 patch 点，因此 shared logging 接入不改变既有 Web 回归契约。

### 5.3 触发场景覆盖

| 触发场景 | 是否写日志 |
|----------|-----------|
| 外部 API `/external/verification-code` | ✅ |
| 主应用前端 `/api/emails/<email>/extract-verification` | ✅ |
| 主应用前端 `/api/temp-emails/<email>/extract-verification` | ✅ |
| 邮箱池 claim 后自动等待验证码 | ✅（若有提取动作）|
| 内部定时刷新（非验证码提取场景） | ❌ |

### 5.4 三条线路与总控板可见性

| 线路 | 当前提取接口 | 是否写 `verification_extract_logs` | 统一监控面板当前可见性 |
|------|-------------|-----------------------------------|------------------------|
| 浏览器 AIUI / 扩展伴生面板 | `/api/external/verification-code` / `/api/external/verification-link` | ✅ | **能看到**，但要等总控板下一次重新拉取 |
| 对外 API 调用方 | `/api/external/verification-code` / `/api/external/verification-link` | ✅ | **能看到**，但要等总控板下一次重新拉取 |
